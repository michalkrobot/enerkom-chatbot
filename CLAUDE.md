## What this is

A RAG chatbot for a non-profit's website. A nightly **indexer** crawls the site
and ingests documents into **PostgreSQL + pgvector**; an **API** answers visitor questions
by retrieving relevant chunks and calling **Azure OpenAI**, returning a streamed answer with
citations. A **TypeScript widget** is embedded on the host site via a single `<script>` tag.

Hard requirements (see [`docs/00-architecture.md`](docs/00-architecture.md)):
- **~Zero running cost** — Azure OpenAI free/cheap tier + an existing shared Postgres.
- **Czech** — system prompt and UI are Czech; the embedding model is multilingual.
- **No hallucination** — answer only from retrieved context; otherwise say "nevím" (I don't
  know) + contact, and always return citations.
- **GDPR** — public content only; do not log user questions with personal data.

## Architecture

Layered / component-based with a shared kernel (NOT VSA/Clean/DDD — the domain is
retrieval-centric, low business-rule complexity). Three runtimes over one shared library:

| Project | Type | Responsibility |
|---|---|---|
| `EnerkomChatbot.Core` | class library (`Microsoft.NET.Sdk`) | models, abstractions, RAG services, embeddings/chat clients, `PgVectorStore`. **No HTTP.** |
| `EnerkomChatbot.Indexer` | worker / console (`Microsoft.Extensions.Hosting`) | offline batch: load → chunk → embed → upsert (mark-and-sweep). Scheduled (cron, ~1×/day). |
| `EnerkomChatbot.Api` | ASP.NET Core Minimal API (`Microsoft.NET.Sdk.Web`) | online: `POST /api/chat` (SSE), `GET /health`. Retrieval + LLM call. |
| `web/widget` | TypeScript + Vite | embeddable Shadow-DOM widget, mascot "Elektron". |

**Dependency rule:** `EnerkomChatbot.Indexer → EnerkomChatbot.Core` and `EnerkomChatbot.Api → EnerkomChatbot.Core`.
`EnerkomChatbot.Core` must never reference `Api` or `Indexer`. Everything shared belongs in Core.

Boundary invariants (do not violate):
- Indexer knows no HTTP — pure batch over Core.
- API never crawls or indexes synchronously in a request handler — it only reads the DB + calls the LLM.
- The embeddings client lives in Core — Indexer and API **must** use the same model and vector dimension (1536), or retrieval breaks.
- The widget never holds API keys — all Azure OpenAI traffic goes through `EnerkomChatbot.Api`.

See [`docs/01-project-structure.md`](docs/01-project-structure.md) for the full folder layout.

## Tech stack

- **TFM** `net10.0` · **C#** latest · **Nullable** enable · **ImplicitUsings** enable · SDK 10.0.101
- **Build:** `.slnx` solution, `Directory.Build.props` (shared props), `Directory.Packages.props` (central package versions)
- **Database:** PostgreSQL + **pgvector** (`vector(1536)`, HNSW index) via `Npgsql` + `Pgvector`
- **AI:** Azure OpenAI (`gpt-4.1-mini` chat, `text-embedding-3-small` embeddings) via
  `Azure.AI.OpenAI` + `Microsoft.Extensions.AI.OpenAI` → expose as `IChatClient` /
  `IEmbeddingGenerator`. Our `EnerkomChatbot.Core/Abstractions` interfaces are thin wrappers over M.E.AI.
- **Resilience:** `Microsoft.Extensions.Http.Resilience` — retry/backoff on 429 (TPM) and 5xx.
- **Auth:** none for end users (public widget). Azure OpenAI auth via API key or
  `DefaultAzureCredential` (managed identity preferred in prod). No app-level user auth.
- **Caching / messaging:** none.
- **Indexer parsing:** `AngleSharp` (HTML), `UglyToad.PdfPig` (PDF), `DocumentFormat.OpenXml` (DOCX).
- **Testing:** xUnit v3 + `Microsoft.AspNetCore.Mvc.Testing` (WebApplicationFactory) +
  Testcontainers for real Postgres+pgvector integration tests.

## Conventions

- One public type per file; file name = type name.
- Async methods end in `Async` and take a `CancellationToken`.
- No business logic in `Program.cs` — wiring only (DI, endpoints).
- Configuration via the Options pattern (`IOptions<AzureOpenAIOptions>`, `IOptions<IndexerOptions>`).
- Config keys (must match across API/Indexer): `Database__ConnectionString`,
  `AzureOpenAI__Endpoint`, `AzureOpenAI__ApiKey`, `AzureOpenAI__ChatDeployment`,
  `AzureOpenAI__EmbeddingDeployment`, `Cors__AllowedOrigins__*`.
